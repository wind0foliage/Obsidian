# 销售出库单：接口下单和订单递交成功业务流程分析

## 一、业务流程概述

销售出库单中，"接口下单"和"订单递交成功"是两个关键的业务流程节点：

- **接口下单（type=5）**：ERP系统通过接口推送订单到WMS系统，订单被成功创建
- **订单递交成功（type=7）**：订单从"等递交"状态（status=10）递交到"待客审"状态（status=30），完成订单的前置处理

---

## 二、接口下单业务流程（type=5）

### 2.1 核心存储过程

**存储过程名称**：`SP_API_PUSH_TRADE`  
**文件位置**：`WdtWMS/db/sp.sql`  
**起始行号**：47054

### 2.2 业务流程步骤

#### 步骤1：参数接收与初始化
- **输入参数**：
  - `P_OwnerId`：货主ID（原货主）
  - `P_SourceId`：订单来源ID（大于0表示接口下发）
  - `P_TradeNo`：订单编号
- **初始化变量**：虚拟货主管理、仓库信息、订单明细游标等

#### 步骤2：虚拟货主映射
```sql
-- 判断是否使用虚拟货主
CALL SP_UTILS_GET_CFG('virtual_owner_manage', V_VirtualOwnerManage, 0);
-- 如果使用虚拟货主，将原货主映射到虚拟货主
SELECT svor.owner_id INTO V_VirtualOwnerId 
FROM sys_virtual_owner_relationship svor 
WHERE svor.origin_owner_id = V_OriginOwnerId;
```

#### 步骤3：订单重复性校验
- 检查订单是否已存在（相同订单号、货主、来源）
- 如果已存在，设置错误信息："订单已存在"
- 检查是否在其他订单类型中已存在（TMS订单、B2B订单）

#### 步骤4：仓库权限校验
```sql
-- 校验仓库是否存在、是否停用、货主是否有权限
UPDATE api_sales_trade tst
  LEFT JOIN sys_warehouse sw ON sw.warehouse_no=tst.warehouse_code
  LEFT JOIN cfg_owner_warehouse sow ON sow.warehouse_id=sw.warehouse_id
SET tst.error_message = CASE
  WHEN sw.warehouse_id IS NULL THEN CONCAT('仓库',tst.warehouse_code,'不存在')
  WHEN sw.is_disabled<>0 THEN CONCAT('仓库',tst.warehouse_code,'已停用')
  WHEN sow.warehouse_id IS NULL THEN CONCAT('该货主无此仓库权限 ',tst.warehouse_code)
  ELSE '' END
```

#### 步骤5：货品校验与映射
- 根据商家编码（spec_no）查找货品规格ID（spec_id）
- 校验货品是否存在、是否匹配
- 如果使用虚拟货主，映射虚拟货品

#### 步骤6：地址信息校验
```sql
-- 校验收件人信息
UPDATE api_sales_trade
SET error_message = CASE
  WHEN receiver_mobile='' AND receiver_telno='' THEN '收件人电话不能为空'
  WHEN receiver_area='' OR receiver_address='' THEN '收件人地址不能为空'
  WHEN logistics_code='' AND logistics_no<>'' THEN '指定了物流单号未指定物流公司'
  ELSE '' END
```

#### 步骤7：库存校验（可选）
根据配置 `under_stock_not_allow_push_type` 决定是否校验库存：
- 如果开启库存校验，检查可发库存是否充足
- 不足时设置错误信息并回滚

#### 步骤8：订单插入
```sql
-- 生成订单系统编号
SET V_TradeSysNo = FN_SYS_NO('trade');

-- 插入销售订单主表（sales_trade）
INSERT INTO sales_trade(...)
SELECT ... FROM api_sales_trade WHERE error_message='' AND rec_id = V_RecId;

-- 订单状态初始化为 10（等递交）
trade_status = 10
```

#### 步骤9：订单明细插入
```sql
-- 遍历订单明细游标
OPEN detail_cursor;
DETAIL_LABEL:LOOP
  FETCH detail_cursor INTO ...;
  -- 插入订单明细到 sales_trade_order 表
  INSERT INTO sales_trade_order(...);
  -- 更新待审核库存 stock_checking_spec
END LOOP;
CLOSE detail_cursor;
```

#### 步骤10：订单扩展信息处理
- 插入收件人加密数据到 `sales_trade_encrypt_data_ex`
- 插入订单扩展信息到 `sales_trade_ex`（国际物流、包装、加密信息等）
- 插入原始单号到 `sales_trade_src_tids`

#### 步骤11：日志记录
```sql
-- 插入"接口下单"日志（type=5）
INSERT INTO sales_trade_log(trade_id, operator_id, `type`, message, created) 
SELECT V_TradeId, 0, 5, '接口下单', V_Now 
FROM api_sales_trade tst
WHERE tst.error_message = '' AND tst.rec_id = V_RecId;
```
**日志位置**：`WdtWMS/db/sp.sql:47989`

#### 步骤12：订单状态同步
```sql
-- 订单创建，回传操作节点
CALL I_ORDER_STATUS_SYNC(V_TradeId, 0, 1);
```

#### 步骤13：事务提交与清理
- 提交事务
- 删除临时接口表数据（`api_sales_trade`、`api_sales_order`）
- 返回结果给ERP系统

### 2.3 关键数据表

- **接口接收表**：
  - `api_sales_trade`：接口订单主表
  - `api_sales_order`：接口订单明细表

- **正式业务表**：
  - `sales_trade`：销售订单主表（status=10：等递交）
  - `sales_trade_order`：销售订单明细表
  - `sales_trade_log`：订单操作日志表（type=5：接口下单）
  - `sales_trade_src_tids`：原始单号表
  - `stock_checking_spec`：待审核库存表

### 2.4 订单状态变化

```
接口接收 → 订单创建 → trade_status = 10（等递交）
```

## 三、订单递交成功业务流程（type=7）

### 3.1 核心存储过程

**存储过程名称**：`I_SALES_DELIVER_TRADE`  
**文件位置**：`WdtWMS/db/sp.sql`  
**起始行号**：8905  
**注释**：递交订单

### 3.2 业务流程步骤

#### 步骤1：参数接收与初始化
- **输入参数**：
  - `P_TradeID`：订单ID
  - `P_OperatorID`：操作员ID
- 读取订单信息（货主、仓库、物流、收件人信息等）

#### 步骤2：订单状态校验
```sql
-- 订单状态必须是 10（等递交）
IF V_TradeStatus <> 10 THEN
  ROLLBACK;
  SET @sys_code = 2, @sys_message = '订单状态不正确';
  LEAVE MAIN_LABEL;
END IF;
```

#### 步骤3：超时罚款时间计算
```sql
-- 读取平台超时罚款间隔
SELECT overtime INTO V_finedTimeInterval 
FROM platform_overtime 
WHERE platform_id = V_PlatformId;

-- 计算超时罚款时间 = 付款时间 + 超时罚款间隔
IF V_PayTime <> '0000-00-00 00:00:00' THEN
  SET V_OvertimeFinedTime = DATE_ADD(V_PayTime, INTERVAL V_finedTimeInterval HOUR);
END IF;
```

#### 步骤4：地址重新匹配
- 如果省市区ID为0，则通过地址文本模糊匹配
- 更新省市区ID到订单表

#### 步骤5：订单拆分
```sql
-- 拆分订单（必须在选择物流和包装之前）
CALL I_DL_SPLIT_TRADE(P_TradeId, P_OperatorID);
```

#### 步骤6：虚拟仓选择
```sql
-- 根据订单信息和策略决定最终仓库
CALL I_DECIDE_WAREHOUSE(P_TradeID, V_WarehouseId, P_OperatorID);
```

#### 步骤7：货品信息刷新
- 更新订单明细的货品名称、规格、重量、体积等信息
- 计算订单总货品种类、总数量、总重量、总体积、总金额
- 计算货品聚合码（goods_aggregate）
- 更新订单标记（flag_id）

#### 步骤8：包装选择
根据包装策略配置选择包装：
```sql
-- 根据订单组成的包装策略（优先执行）
IF @cfg_open_package_strategy_priority_order_composition THEN
  CALL I_DL_DECIDE_PACKAGE(V_PackageID, P_TradeID, V_Volume, V_Weight, 2);
END IF;

-- 其他包装策略（根据重量或体积或冷链）
IF @cfg_open_package_strategy_priority_other THEN
  CALL I_DL_DECIDE_PACKAGE(V_PackageID, P_TradeID, V_Volume, V_Weight, @cfg_open_package_strategy_type);
END IF;
```

#### 步骤9：物流选择
```sql
-- 如果物流未指定，自动选择物流
IF V_LogisticsId = 0 THEN
  CALL I_DL_DECIDE_LOGISTICS(V_LogisticsId, P_TradeID);
END IF;

-- 更新物流ID到订单
UPDATE sales_trade SET logistics_id = V_LogisticsId WHERE trade_id = P_TradeID;
```

#### 步骤10：地区别名匹配
```sql
-- 匹配物流地区别名
CALL I_DL_DECIDE_AREA_ALIAS(V_AreaAlias, V_LogisticsId, 0, V_ReceiverProvince, V_ReceiverCity, V_ReceiverDistrict);
```

#### 步骤11：预估邮资成本
```sql
-- 如果选择了物流，计算预估邮资成本
IF V_LogisticsId > 0 THEN
  CALL I_UPDATE_POST_COST(P_TradeID, V_Weight, 1, 1);
END IF;
```

#### 步骤12：打印备注处理
- 从客服备注提取打印备注（如果配置开启）
- 根据配置决定是否在打印备注前添加货品汇总信息

#### 步骤13：更新订单信息
```sql
UPDATE sales_trade
SET trade_status     = 30,  -- 更新为"待客审"状态
    logistics_id     = V_LogisticsId,
    print_remark     = V_PrintRemark,
    receiver_dtb     = IF(FN_EMPTY(V_LogisticsNO), V_AreaAlias, receiver_dtb),
    goods_type_count = V_GoodsTypeCount,
    goods_count      = V_GoodsCount,
    weight           = V_Weight,
    volume           = V_Volume,
    goods_amount     = V_GoodsAmount,
    package_id       = V_PackageID,
    overtime_fined_time = V_OvertimeFinedTime
WHERE trade_id = P_TradeID;
```

#### 步骤14：日志记录
```sql
-- 插入"订单递交成功"日志（type=7）
INSERT INTO sales_trade_log(trade_id, operator_id, TYPE, message) 
VALUES(P_TradeID, P_OperatorID, 7, '订单递交成功');
```
**日志位置**：`WdtWMS/db/sp.sql:9283`

#### 步骤15：提交事务
```sql
COMMIT;
```

#### 步骤16：自动审核（可选）
根据配置 `deliver_trade_auto_check` 决定是否自动审核：
```sql
-- 如果开启自动审核
IF @cfg_deliver_trade_auto_check = 1 THEN
  -- 检查是否在自动审核时间范围内
  -- 检查是否开启多级审核
  -- 调用审核存储过程
  CALL I_SALES_TRADE_CHECK(P_TradeID, 1, P_OperatorID, 0);
END IF;
```

#### 步骤17：自动分配货位（可选）
根据配置 `check_trade_auto_allocate` 决定是否自动分配货位：
```sql
-- 如果开启自动分配货位
IF @cfg_check_trade_auto_allocate = 1 THEN
  -- 判断是否为相同货品订单
  CALL I_JUDGE_SAME_GOODS_ORDER(V_StockoutId, V_SameGoodsAllocate);
  
  -- 根据订单类型分配货位
  IF V_SameGoodsAllocate THEN
    CALL I_SAME_ORDER_DISTRIBUTION_POSITION(...);
  ELSE
    CALL I_DISTRIBUTION_POSITION(...);
  END IF;
END IF;
```

#### 步骤18：店铺自动创建
```sql
-- 如果店铺不存在，自动创建店铺
IF V_ShopNo='' AND V_OldShopNo='' THEN
  SET V_ShopNo = FN_SYS_NO('shop');
  INSERT IGNORE INTO sys_shop (owner_id, shop_no, shop_name, create_method) 
  VALUES (V_OwnerId, V_ShopNo, V_ShopName, 1);
END IF;
```

### 3.3 关键数据表

- **订单主表**：
  - `sales_trade`：订单状态从 10（等递交）→ 30（待客审）

- **日志表**：
  - `sales_trade_log`：订单操作日志表（type=7：订单递交成功）

- **出库单表**（如果生成）：
  - `stockout_order`：销售出库单表

### 3.4 订单状态变化

```
等递交（status=10） → 递交处理 → 待客审（status=30）
```

### 3.5 关键配置项

1. **自动审核配置**：
   - `deliver_trade_auto_check`：递交订单时自动审核订单（0：关闭，1：开启）
   - `sales_auto_check_begin`：自动审核开始时间
   - `sales_auto_check_end`：自动审核结束时间
   - `trade_multi_check_step`：多级审核步数（大于1时不自动审核）

2. **自动分配货位配置**：
   - `check_trade_auto_allocate`：订单审核自动分配货位（0：关闭，1：开启）
   - `same_goods_order_allocate_stock_zone_type`：相同货品订单分配货位策略

3. **包装策略配置**：
   - `open_package_strategy_priority_order_composition`：根据订单组成的包装策略
   - `open_package_strategy_priority_other`：其他包装策略
   - `open_package_strategy_priority_other_type`：包装策略类型（0：重量，1：体积，2：冷链）

4. **物流匹配配置**：
   - `logistics_match_mode`：物流匹配策略模式

## 四、业务流程关联

### 4.1 流程顺序

```
接口下单（type=5）
    ↓
订单状态：10（等递交）
    ↓
订单递交（type=7）
    ↓
订单状态：30（待客审）
    ↓
订单审核（type=55）
    ↓
订单状态：55（已确认）
```

### 4.2 调用关系

**接口下单调用链**：
```
ERP系统 → API接口 → SP_API_PUSH_TRADE → sales_trade（status=10）
```

**订单递交调用链**：
```
用户操作/定时任务 → I_SALES_DELIVER_TRADE → sales_trade（status=30）
                                     ↓
                            可选：I_SALES_TRADE_CHECK（自动审核）
                                     ↓
                            可选：I_DISTRIBUTION_POSITION（自动分配货位）
```

### 4.3 C#调用点

在C#代码中，订单递交可能通过以下方式调用：
- 订单管理界面（`TradeMgrWindow.cs`）的手动递交操作
- 批量递交操作
- 定时任务自动递交

## 六、异常处理

### 6.1 接口下单异常

常见错误信息：
- "订单已存在"：相同订单号已存在
- "货品不存在"：商家编码在系统中不存在
- "仓库不存在"或"仓库已停用"：仓库校验失败
- "该货主无此仓库权限"：权限校验失败
- "收件人电话不能为空"：地址信息不完整
- "可发库存不足"：库存校验失败（如果开启）
- "接口超时"：处理时间超过20秒

### 6.2 订单递交异常

常见错误信息：
- "订单状态不正确"：订单状态不是10（等递交）
- "订单物流没选"：物流选择失败
- "订单拆分失败"：订单拆分过程出错
- "相同货品订单递交成功并自动分配货位失败"：货位分配失败
- "订单递交成功并自动分配货位失败"：货位分配失败

## 九、总结

### 9.1 接口下单流程特点

1. **异步处理**：ERP通过接口推送，WMS异步处理
2. **严格校验**：多层校验确保数据完整性（仓库、货品、地址、库存等）
3. **状态初始化**：订单创建后状态为10（等递交）
4. **日志记录**：记录"接口下单"日志（type=5）

### 9.2 订单递交流程特点

1. **状态转换**：从"等递交"（10）转换为"待客审"（30）
2. **策略处理**：包含订单拆分、虚拟仓选择、包装选择、物流选择等策略
3. **自动化**：支持自动审核、自动分配货位
4. **日志记录**：记录"订单递交成功"日志（type=7）

### 9.3 业务流程完整性

这两个流程构成了订单从ERP系统进入到WMS系统，再到可以审核出库的完整链路，是WMS系统的核心业务流程之一。

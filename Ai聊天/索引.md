# 一、先搞懂：索引到底是什么（原理篇）

## 1. 索引本质

- 索引是**排好序的快速查找数据结构**
    
- 目的：**减少磁盘IO，避免全表扫描**
    

## 2. InnoDB 索引结构

- 底层：**B+ 树**
    
- 只有叶子节点存**完整数据**（主键索引）
    
- 非叶子节点只存键值，用来快速定位
    

## 3. 两种核心索引

1. **聚簇索引（主键索引）**
    
    1. 叶子节点存**整行数据**
        
    2. 每张表**必有且只有一个**
        
2. **二级索引（普通/唯一索引）**
    
    1. 叶子节点存**主键值**
        
    2. 查数据需要**回表**
        

## 4. 回表 & 覆盖索引

- **回表**：二级索引查到主键 → 再去主键索引查数据
    
- **覆盖索引**：查询的字段**全都在索引里**，不需要回表
    

→ 性能最高

# 二、必须掌握：索引分类（基础篇）

1. **主键索引 PRIMARY KEY**
    
    1. 唯一、非空、聚簇索引
        
2. **唯一索引 UNIQUE KEY**
    
    1. 业务唯一，允许 NULL（一个）
        
3. **普通索引 INDEX**
    
    1. 只加速，不约束
        
4. **联合索引（最关键）**
    
    1. 多个字段组成一个索引
        
    2. 遵循**最左前缀原则**
        

---

# 三、核心规则：联合索引怎么建（进阶篇）

## 1. 最左前缀原则（死规则）

索引 `(a,b,c)`

- 能用到：`a` / `a,b` / `a,b,c`
    
- 用不到：`b` / `c` / `b,c`
    

## 2. 联合索引字段顺序黄金法则

1. **等值查询（=、IN）放前面**
    
2. **范围查询（>、<、between、like）放最后**
    
3. **区分度高的放前面**
    

示例：

```SQL
where user_id=? and status=? and create_time>?...
索引：(user_id, status, create_time)
```

## 3. IN 等价于 =

- IN 可以放在联合索引**最左**
    
- 最左字段**不能省略**，宁可 IN(全部)，也不能不写

# 四、索引实战：怎么建索引（实战篇）

## 1. 必须建索引的字段

- WHERE 里经常出现的字段
    
- JOIN 关联字段
    
- ORDER BY / GROUP BY 字段
    

## 2. 不要建索引的字段

- 区分度极低：性别、状态（0/1）
    
- 很少查询的字段
    
- 小表（数据量 < 1000）
    

## 3. 建索引最佳实践

- 单表索引**不超过 5 个**
    
- 联合索引**优于**多个单列索引
    
- 优先做**覆盖索引**

# 五、怎么判断索引有没有生效（排查篇）

## 1. 看执行计划

```SQL
explain select ...
```

关键字段：

- **type**：最优到最差
    

system > const > eq_ref > ref > range > index > ALL

- **key**：实际使用的索引
    
- **rows**：扫描行数越少越好
    
- **Extra**
    
    - Using index → 覆盖索引（极好）
        
    - Using where → 正常
        
    - Using filesort → 需要优化
        
    - Using temporary → 需要优化
        

## 2. 索引失效常见场景

- 对字段做计算、函数、类型转换
    
- 模糊查询 `%前缀`
    
- 联合索引**跳过最左字段**
    
- 隐式类型转换（varchar = 数字）

# 六、SQL 优化实战（高级篇）

1. **避免 select ***，只查需要字段
    
2. 尽量做成**覆盖索引**
    
3. 大分页优化：
    
    ```SQL
    where id > 10000 limit 10
    ```
    
4. order by 要**遵循索引顺序**
    
5. 禁止在索引字段上使用函数

# 一、先搞懂：索引到底是什么（原理篇）

## 1. 索引本质

- 索引是**排好序的快速查找数据结构**
    
- 目的：**减少磁盘IO，避免全表扫描**
    

## 2. InnoDB 索引结构

- 底层：**B+ 树**
    
- 只有叶子节点存**完整数据**（主键索引）
    
- 非叶子节点只存键值，用来快速定位
    

## 3. 两种核心索引

1. **聚簇索引（主键索引）**
    
    1. 叶子节点存**整行数据**
        
    2. 每张表**必有且只有一个**
        
2. **二级索引（普通/唯一索引）**
    
    1. 叶子节点存**主键值**
        
    2. 查数据需要**回表**
        

## 4. 回表 & 覆盖索引

- **回表**：二级索引查到主键 → 再去主键索引查数据
    
- **覆盖索引**：查询的字段**全都在索引里**，不需要回表
    

→ 性能最高

---

# 二、必须掌握：索引分类（基础篇）

1. **主键索引 PRIMARY KEY**
    
    1. 唯一、非空、聚簇索引
        
2. **唯一索引 UNIQUE KEY**
    
    1. 业务唯一，允许 NULL（一个）
        
3. **普通索引 INDEX**
    
    1. 只加速，不约束
        
4. **联合索引（最关键）**
    
    1. 多个字段组成一个索引
        
    2. 遵循**最左前缀原则**

# 三、核心规则：联合索引怎么建（进阶篇）

## 1. 最左前缀原则（死规则）

索引 `(a,b,c)`

- 能用到：`a` / `a,b` / `a,b,c`
    
- 用不到：`b` / `c` / `b,c`
    

## 2. 联合索引字段顺序黄金法则

1. **等值查询（=、IN）放前面**
    
2. **范围查询（>、<、between、like）放最后**
    
3. **区分度高的放前面**
    

示例：

```SQL
where user_id=? and status=? and create_time>?...
索引：(user_id, status, create_time)
```

## 3. IN 等价于 =

- IN 可以放在联合索引**最左**
    
- 最左字段**不能省略**，宁可 IN(全部)，也不能不写

# 四、索引实战：怎么建索引（实战篇）

## 1. 必须建索引的字段

- WHERE 里经常出现的字段
    
- JOIN 关联字段
    
- ORDER BY / GROUP BY 字段
    

## 2. 不要建索引的字段

- 区分度极低：性别、状态（0/1）
    
- 很少查询的字段
    
- 小表（数据量 < 1000）
    

## 3. 建索引最佳实践

- 单表索引**不超过 5 个**
    
- 联合索引**优于**多个单列索引
    
- 优先做**覆盖索引**
    

## 4. 单表数据增长预测（索引动态调整前提）

核心目的：根据预测结果调整索引（如拆分联合索引、新增索引、删除冗余索引）、优化表结构（如分表），避免数据量激增导致索引失效、查询变慢。以下是 4 种常规且实用的方法，从简单到复杂逐步递进：

### （1）历史数据统计法（最基础、最常用）

- 核心逻辑：基于过去一段时间的历史数据，计算日均/周均/月均增长数量，推算未来增长趋势（适用于业务稳定、增长平稳的表，如用户表、订单表）。
    
- 操作步骤：
    
    - 通过 SQL 统计历史数据量：`select count(id) as total from 表名 where create_time between '起始时间' and '结束时间';`
        
    - 计算时间跨度（如 30 天）和总增长量，得出日均增长量 = 总增长量 / 时间天数。
        
    - 预测未来：如当前数据量 100 万，日均增长 5000 条，预测 6 个月（180 天）后数据量 = 100 万 + 5000×180 = 190 万。
        
- 注意：需排除异常数据（如活动期间的突发增长），取正常业务周期的平均值。
    

### （2）趋势拟合分析法（适用于有波动但有规律的表）

- 核心逻辑：收集过去 N 个周期（如每月、每周）的期末数据量，用简单拟合（线性、指数）分析增长趋势，预测未来数据量。
    
- 常用工具：Excel（插入趋势线、使用 FORECAST 函数）、Python（pandas 拟合、matplotlib 可视化），无需复杂算法，新手可快速上手。
    
- 适用场景：业务有淡旺季（如电商订单表、活动表），增长不是匀速，但有明显规律。
    

### （3）业务场景倒推法（最贴合实际，优先推荐）

- 核心逻辑：不单纯看历史数据，结合业务规划（如用户增长目标、活动计划、业务扩张）倒推数据增长，更精准。
    
- 示例：
    
    - 业务规划：未来 3 个月新增用户目标 10 万，每个用户平均生成 5 条订单数据。
        
    - 当前订单表数据量 80 万，预测 3 个月后订单数据量 = 80 万 + 10 万×5 = 130 万。
        
- 关键：需和业务部门确认规划（如市场、运营），避免脱离实际业务的盲目预测。
    

### （4）数据库监控工具法（实时动态预测）

- 核心逻辑：通过数据库监控工具，实时采集表数据量、插入频率，自动生成增长趋势图，支持短期（如1个月）精准预测。
    
- 常用工具：MySQL 自带的 INFORMATION_SCHEMA（定期查询表行数）、Navicat 监控、Prometheus + Grafana（可视化监控，适合中大型项目）。
    
- 优势：无需手动统计，实时更新，可及时发现异常增长（如数据泄露、程序bug导致的批量插入）。
    

补充注意：预测结果需留 20%-30% 冗余（如预测 190 万，按 240 万规划索引和表结构），避免突发增长导致性能问题；同时每 1-2 个月复盘一次，修正预测模型。

# 五、怎么判断索引有没有生效（排查篇）

## 1. 看执行计划

```SQL
explain select ...
```

关键字段：

- **type**：最优到最差
    
- system > const > eq_ref > ref > range > index > ALL
    
- **key**：实际使用的索引
    
- **rows**：扫描行数越少越好
    
- **Extra**
    
    - Using index → 覆盖索引（极好）
        
    - Using where → 正常
        
    - Using filesort → 需要优化
        
    - Using temporary → 需要优化
        

## 2. 索引失效常见场景

- 对字段做计算、函数、类型转换
    
- 模糊查询 `%前缀`
    
- 联合索引**跳过最左字段**
    
- 隐式类型转换（varchar = 数字）

# 六、SQL 优化实战（高级篇）

2. 尽量做成**覆盖索引**

5. 禁止在索引字段上使用函数

# 七、索引底层原理（成神篇）

## 1. B+ 树深层细节

- **页结构**：InnoDB 数据和索引都以“页”为单位存储（默认 16KB），B+ 树的每一层节点对应一个或多个页。
    
- **页分裂**：当主键插入无序（如雪花ID）或页满时，会触发页分裂，拆分出一个新页，导致索引碎片增加、查询IO增多；自增主键可避免频繁页分裂。
    
- **页合并**：当删除数据导致页利用率过低（低于 50%），InnoDB 会将相邻的空闲页合并，减少空间浪费。
    
- **B+ 树高度**：16KB 页可存储约 1000 个索引键值，一般 B+ 树高度为 3-4 层，就能支撑千万级数据（3 层可存 1000×1000×1000 = 10 亿条数据），查询只需 3-4 次磁盘IO。
    

## 2. 索引高级优化机制

- **索引下推（ICP）**：MySQL 5.6+ 新增，将“过滤条件”从服务器层下推到存储引擎层，在回表前过滤无效数据，减少回表次数（例：联合索引 (name, age)，查询 where name like '张%' and age > 20，ICP 会先过滤 age > 20 的数据，再回表）。
    
- **MRR（多范围读取）**：将分散的索引查询结果按主键排序，批量回表，减少磁盘随机IO，提升查询效率（适用于 range 查询、IN 查询）。
    
- **自适应哈希索引（AHI）**：InnoDB 自动对频繁查询的索引键建立哈希索引，将 B+ 树的“范围查找”转为“哈希查找”，提升等值查询速度（无需手动创建，由数据库自动管理）。
    

## 3. 主键选择对索引的影响

- **自增主键（推荐）**：插入有序，避免页分裂，索引结构紧凑，查询效率高；适合大多数业务场景（如 user_id、order_id）。
    
- **雪花ID（分布式场景）**：全局唯一，适合分布式系统，但插入无序，会频繁触发页分裂，产生索引碎片，需定期优化（如重建索引）。
    
- **不推荐用 UUID**：字符串类型，占用空间大，排序效率低，插入无序，页分裂频繁，索引性能差。
    

## 4. 索引碎片与优化

- **碎片产生原因**：频繁插入无序数据、频繁删除数据、页分裂。
    
- **碎片危害**：索引占用空间变大，查询时需要扫描更多页，IO 增加，查询变慢。
    
- **优化方法**：重建索引（ALTER TABLE 表名 ENGINE=InnoDB）、优化表（OPTIMIZE TABLE 表名），适合碎片率高于 30% 的表。

# 八、复杂场景索引设计（精通篇）

## 1. 分表分库下的索引设计

- **分表场景**：水平分表（按时间、按用户ID哈希）时，索引需和分表键配合，避免跨表查询。
    
- **局部索引**：每个分表独立建立索引，只查询当前分表数据，效率高（推荐）。
    
- **全局索引**：适用于跨表查询场景（如全局订单查询），需单独维护一张索引表，记录“索引键-分表标识”，查询时先查索引表，再定位到具体分表。
    
- **注意**：分表后联合索引需包含分表键，避免跨表扫描。
    

## 2. 海量数据（千万+）索引优化技巧

- 减少索引数量：只保留高频查询的索引，避免索引过多导致插入/更新变慢。
    
- 优化联合索引：优先设计覆盖索引，避免回表；将高频等值查询字段放最左，范围查询放最后。
    
- 分区索引：对大表按时间分区（如订单表按月份分区），查询时只扫描目标分区，减少扫描范围。
    
- 避免过度索引：区分度低的字段（如性别）不建索引，小范围查询（如 limit 100）可不用索引。
    

## 3. 特殊场景索引设计

- **全文索引**：适用于模糊查询（如文章标题、内容搜索），MySQL 自带 FULLTEXT 索引，替代 %like% 模糊查询，提升效率。
    
- **空间索引**：适用于地理坐标查询（如附近的门店），使用 SPATIAL 索引，支持距离查询。
    
- **哈希索引**：适用于纯等值查询（如缓存key查询），可手动创建（如用 Redis 辅助），查询速度快于 B+ 树索引。
